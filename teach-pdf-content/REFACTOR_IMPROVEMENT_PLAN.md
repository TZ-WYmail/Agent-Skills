# teach-pdf-content 改进与重构方案

## 1. 文档目的

本文档用于指导 `teach-pdf-content` 的后续分步重构，目标不是只优化网页显示，而是同时解决以下两个根问题：

1. 现有章节笔记很多时候仍然是“教材解压缩版”，不是“可以直接学懂的讲解版”。
2. 现有章节产物过于碎片化，学习者面对的是一包文件，而不是一套清晰的学习流。

因此，后续整改要同时覆盖：

- 内容生成逻辑
- 章节文件结构
- HTML 展示逻辑
- 质量标准与验收方式


## 2. 当前逻辑与主要问题

### 2.1 当前章节产物结构

当前 `teach-pdf-content` 默认按每章生成：

- `00-learning-path.md`
- `01-lesson-notes.md`
- `02-active-recall.md`
- `03-exercises.md`
- `04-glossary.md`
- `05-review-plan.md`
- `06-memory-cards.md`
- `07-code-extracts.md`
- `source-map.md`
- `_meta.json`

课程级还会有：

- `chapter-index.md`

这意味着：

- 对学习者而言，一个章节会出现接近 10 个文件。
- 即使通过 HTML 重新聚合，底层内容仍然是碎片化的。
- 学习者需要自己理解“先看哪个，后看哪个”，负担过大。

### 2.2 现有内容层面的核心问题

当前很多章节小节仍有明显“整理笔记化”倾向，典型模式是：

- 这一节回答什么问题
- 先给结论
- 原文依据
- 直觉理解
- 简单例子

这套模板适合做审校，不足以支撑真正教学。它缺少：

- 为什么这个结论成立
- 这个概念为什么会在这里出现
- 它与前后概念如何连接
- 遍历/算法/结构的状态是如何变化的
- 学生最容易卡在哪里
- 如何从“看懂”过渡到“能自己讲出来”

结果就是：

- 学习者读完笔记，仍需要回教材做第二次“解码”
- 讲解层没有真正替代教材的解释工作
- 网页展示再好，也只是把“半成品内容”展示得更好

### 2.3 当前网页层面的局限

现有 HTML 原型已经解决了这些问题：

- 把文件阅读改成阶段阅读
- 把答案区折叠
- 增加学习模式 / 复习模式
- 增加术语卡片、代码索引、复习时间轴

但它仍有一个边界：

- 网页层只能重排内容，不能凭空补出内容里没有的讲解闭环

因此后续必须把“内容重构”放在“网页强化”前面。


## 3. 重构目标

重构后的 skill 应满足以下目标：

### 3.1 学习体验目标

- 学习者不需要频繁回教材才能理解核心机制
- 每个小节都能独立完成“提出问题 -> 建立直觉 -> 展开机制 -> 解释成立原因 -> 提醒易错点 -> 形成闭卷表达”
- 一个章节应该像一份完整讲义，而不是一个文件集合

### 3.2 文件结构目标

面向学习者的章节主文件收敛为三类：

1. `detailed-notes.md`
2. `practice.md`
3. `review-notes.md`

### 3.3 工程目标

- Markdown 继续作为内容真源
- 机器可读元数据保留，但不再暴露为学习主文件
- HTML 层继续存在，但服务于三类主文件，而不是 8 个文件

### 3.4 知识点导向目标

- 每一章在生成前先抽出“知识点集合”，而不是直接按教材目录写段落
- 每一个小节的讲解深度由“新概念复杂度”决定，而不是所有小节套同一个模板
- 每一章在最开始生成一张“知识点导向的知识图谱”，前端首屏优先展示
- 学习者可以从知识图谱进入讲义，而不是只能从线性目录进入讲义


## 4. 建议的新章节结构

### 4.1 面向学习者的三类主文件

每章只保留三类主文件作为学习入口：

#### A. `detailed-notes.md`

用途：第一次学习与深入理解。

建议合并来源：

- `00-learning-path.md`
- `01-lesson-notes.md`
- `04-glossary.md`
- `07-code-extracts.md`

内部建议结构：

1. 章节定位
2. 先修提醒
3. 这章解决什么问题
4. 最小例子 / 起步例子
5. 章节地图
6. 分节教学正文
7. 关键术语与公式
8. 代码连接与实现陷阱
9. 易混概念对比
10. 闭卷输出模板
11. 10 分钟速读路线

#### B. `practice.md`

用途：做题、自测、巩固、查漏。

建议合并来源：

- `02-active-recall.md`
- `03-exercises.md`

内部建议结构：

1. 使用方式
2. 闭卷解释题
3. 判断/辨析题
4. 过程追踪题
5. Basic / Core / Transfer 练习
6. 提示区
7. 完整答案区
8. 评分要点
9. 常见错误

#### C. `review-notes.md`

用途：二刷、考前压缩、记忆回捞。

建议合并来源：

- `05-review-plan.md`
- `06-memory-cards.md`
- `01-lesson-notes.md` 中的“必须记住/最低完成标准/10 分钟速读”类内容

内部建议结构：

1. 复习路线
2. 今天 / 明天 / 三天后 / 一周后
3. 必记定义与公式
4. 易错点清单
5. 卡片区
6. 3 分钟口头复述模板
7. 错题回炉规则

### 4.2 面向系统的内部文件

以下文件不再作为学习主入口，但可以保留为内部支撑文件：

- `source-map.md`
- `_meta.json`
- `chapter-index.md`
- `knowledge-map.json`

建议定位：

- `source-map.md`：保留给审校、回溯、核对，不在学习入口中强调
- `_meta.json`：纯机器文件
- `chapter-index.md`：课程级组织文件
- `knowledge-map.json`：章节知识图谱的机器文件，用于前端渲染、节点跳转、学习路径推荐

换句话说：

- 学习者只看到 3 个主文件
- 系统和审校仍然可以看到底层元数据


## 5. 内容重构方向：从“整理”升级到“讲解”

### 5.0 先从“章节目录导向”改成“知识点导向”

后续生成逻辑不应再默认按教材小标题直接展开，而应先完成一层知识建模。

每章生成前，先识别并组织以下对象：

- 核心概念
- 先修概念
- 公式与规则
- 算法/过程
- 易混对比点
- 实现锚点
- 练习目标

然后再决定：

- 哪些知识点应该合并讲
- 哪些知识点必须单独讲
- 哪些知识点只需要点到为止
- 哪些知识点必须给 trace、反例、代码连接

换句话说：

- 章节结构不再只是“教材写了 7.4.1，所以我生成 7.4.1”
- 而是“这一节实际包含哪些知识点，这些知识点对初学者有多难，应该怎么讲”

### 5.1 新的小节教学模板

对于数据结构、算法、图、树、递归、状态机这类内容，每个核心小节建议强制采用以下教学模板：

1. 这节为什么会出现在这里
2. 这节到底解决什么问题
3. 先看一个最小例子
4. 观察现象
5. 给正式定义或结论
6. 解释为什么成立
7. 跑一遍过程 / trace
8. 给边界 / 反例 / 非例
9. 跟相邻概念比较
10. 给一句闭卷输出模板

### 5.2 新增“概念复杂度分级”机制

不是每个小节都需要同样长的讲解。后续应先判断“新出现概念”的复杂程度，再决定讲解层级。

建议引入四级复杂度：

#### C1：轻量概念

特点：

- 与已有知识高度相似
- 定义直观
- 不涉及复杂状态变化
- 不需要长过程追踪

适合讲法：

- 1 个最小定义
- 1 个一句话直觉
- 1 个最小例子
- 1 个边界提醒

#### C2：标准概念

特点：

- 需要与已有概念区分
- 会出现在题目中
- 有一定使用条件

适合讲法：

- 问题
- 定义
- 小例子
- 为什么这样定义
- 易混点
- 闭卷说法

#### C3：复杂概念

特点：

- 需要过程追踪
- 容易误解
- 依赖前面多个知识点
- 与算法过程或结构机制绑定

适合讲法：

- 问题场景
- 最小图/树/输入例子
- 分步 trace
- 为什么成立
- 易错点
- 相邻概念比较
- 代码连接
- 闭卷模板

#### C4：高复杂度概念

特点：

- 高抽象
- 有递推/证明/多层依赖
- 常常是全章难点
- 学生只看定义几乎不可能懂

适合讲法：

- 引入问题
- 最小反例或最小例子
- 分层解释
- trace
- 为什么成立
- 为什么不能换一种做法
- 对比多个相邻概念
- 题型映射
- 代码/公式双连接
- 闭卷表达模板
- 复习压缩版

### 5.3 概念复杂度的判定维度

建议用以下维度给知识点打分：

- 新颖度：是否是学习者第一次正式接触
- 抽象度：是否难以直观看懂
- 依赖深度：是否依赖多个前置知识点
- 动态性：是否需要过程追踪
- 易错度：是否高频混淆
- 教材歧义度：教材本身是否解释不足

可以把每个维度简单打 `0/1/2`，总分决定 C1-C4：

- `0-2` -> C1
- `3-5` -> C2
- `6-8` -> C3
- `9-12` -> C4

这套机制的作用是：

- 避免所有知识点都被平均讲解
- 把篇幅集中给真正难的部分
- 让“详细笔记”更像一份老师写出来的讲义，而不是平均长度的摘要集合

### 5.4 需要强制补齐的内容类型

后续生成逻辑中，以下内容必须显式产出，不能只给结论：

- 原理成立原因
- 状态变化过程
- 为什么不会出错
- 为什么不会遗漏
- 为什么这一步是必要的
- 与相邻概念的边界差异

### 5.5 对“结构/算法类小节”的刚性要求

如果某小节主题是以下任意类型：

- 树 / 图 / 递归 / 回溯
- 遍历 / 最短路 / 最小生成树
- 并查集 / 状态树 / Huffman / KMP
- 公式推导 / 编号规则 / 存储结构

则必须至少包含：

- 一个最小例子
- 一个过程追踪
- 一个“为什么成立”的解释
- 一个易错点
- 一个闭卷说法

### 5.6 小节生成的新流程

建议未来每个小节的生成按下面顺序走：

1. 从原文抽取候选知识点
2. 标记哪些是“本节新出现的概念”
3. 为每个新概念做复杂度分级
4. 合并为本节的“主知识点”和“辅知识点”
5. 决定本节讲解模板深度
6. 生成正文
7. 生成闭卷表达
8. 生成练习映射


## 6. 以图论章节为例的内容整改标准

### 6.1 当前问题示例

类似下面这样的段落：

- 这一节回答什么问题
- 先给结论
- 原文依据
- 直觉理解
- 简单具体例子

如果没有进一步解释：

- 为什么 DFS/BFS 一次恰好得到一个连通分量
- 为什么遍历留下的边恰好构成树
- 为什么非连通图会得到森林

那这段就仍然是“摘要”，不是“讲解”。

### 6.2 目标写法示例思路

以“连通分量、生成树与生成森林”为例，目标不是只写：

- 从一个未访问点发起 DFS/BFS，会得到一个连通分量

而要补齐：

1. 为什么只能走到同一块里的点
2. 为什么不会漏掉这块里的点
3. 为什么第一次到达新点时留下的边不会成环
4. 为什么这些边能把访问点连起来
5. 为什么这不是“最小生成树”

这类解释一旦补齐，学习者才可能不回教材也学懂。

## 7. 章节知识图谱方案

### 7.1 目标

每一章在最开始生成一张“知识点导向的知识图谱”，并在前端界面首屏展示。

这张图谱不是装饰图，而是学习入口图。它的作用是：

- 帮学习者先看清这章在讲什么
- 帮学习者知道知识点之间的依赖关系
- 帮学习者知道哪些点是核心、哪些点是扩展、哪些点最难
- 帮前端做节点点击跳转和推荐学习路径

### 7.2 图谱节点类型

建议把知识点分成以下节点类型：

- `core_concept`：核心概念
- `prerequisite`：先修点
- `procedure`：算法/过程
- `formula`：公式/规则
- `comparison`：易混辨析点
- `implementation`：代码/实现锚点
- `application`：应用场景

### 7.3 图谱边类型

建议至少支持这些关系：

- `requires`：依赖
- `explains`：解释
- `contrasts`：对比
- `implements`：代码实现
- `applies_to`：应用到
- `reviews_with`：复习时一起回看

### 7.4 节点元数据

每个节点建议至少包含：

- `id`
- `title`
- `type`
- `complexity_level`
- `importance_level`
- `summary`
- `section_anchor`
- `related_exercises`
- `related_review_points`

### 7.5 图谱生成方式

建议为每章额外生成：

- `knowledge-map.json`

用于前端直接渲染。

同时，在 `detailed-notes.md` 顶部保留一段“知识图谱说明”，便于 Markdown 自身也能表达：

- 本章核心点有哪些
- 学习顺序是什么
- 最难点是什么

### 7.6 前端展示建议

知识图谱应放在章节页最开始，位置优先级高于普通目录。

建议首屏结构：

1. 章节标题
2. 一句话章节主问题
3. 知识图谱
4. 推荐学习路径
5. 再进入详细笔记

前端交互建议：

- 点击节点高亮相关节点
- 点击节点跳到对应小节
- 按复杂度筛选节点
- 按“学习模式 / 复习模式”切换图谱强调内容
- 首次默认高亮“推荐起步路径”

### 7.7 图谱与三类主文件的关系

图谱不是独立学习文件，而是三类主文件的导航层。

它应驱动：

- `detailed-notes.md` 的阅读顺序
- `practice.md` 的题目映射
- `review-notes.md` 的复习聚合

换句话说，知识图谱应成为整章的统一索引。


## 8. 网页层的下一步优化方向

在三类主文件收敛完成后，HTML 层可以进一步强化为真正的“教学网页”，而不是 Markdown 浏览器。

### 7.1 页面结构建议

三类主文件分别映射为：

- 详细笔记页
- 练习页
- 复习页

而不是继续把 8 个文件分别渲染成多个独立区块。

### 7.2 详细笔记页建议

详细笔记页应支持：

- 小节分层阅读
- 最小例子高亮
- trace 分步展开
- “为什么成立”卡片
- 易错点卡片
- 代码/图示联动

### 7.3 练习页建议

练习页应支持：

- 题目按类型切换
- 先题后答
- 难度过滤
- 目标点过滤
- 答案延迟显示
- 错误类型提示

### 7.4 复习页建议

复习页应支持：

- 速读模式
- 卡片模式
- 错题回炉模式
- 口头复述模式
- 必记公式/定义清单

### 7.5 现有 HTML 原型的定位

现有 renderer 已经验证了一个重要方向：

- Markdown 内容源 + 静态 HTML 阅读层

这个方向是正确的，后续不需要推翻，只需要把输入结构从“8 文件包”切换为“3 文件主包”，并加强页面语义。


## 9. Skill 规则层面的整改建议

### 8.1 输出标准从“文件完整”改成“教学完整”

后续质量检查不能只看有没有生成齐文件，更要看是否完成教学闭环。

### 8.2 建议新增的 P0 规则

以下情况建议视为 P0 失败：

- 一个结构/算法小节没有最小例子
- 一个结构/算法小节没有过程追踪
- 小节给了结论但没有解释为什么成立
- 学习者读完后仍必须回教材才能理解核心机制
- 只有原文改写，没有教师解码
- 一个章节主文件仍主要由定义表和摘抄表组成

### 8.3 建议新增的 P1 规则

- 小节之间缺少过渡
- 易错点只是罗列，没有解释误因
- 公式只给出形式，没有解释符号和适用条件
- 代码块存在，但没有说明它在概念上对应什么
- 复习页没有明确“先复习什么，后复习什么”


## 10. 分步重构方案

建议分四个阶段推进，不要一次大改完。

### 阶段一：定义新产物模型

目标：

- 固化三类主文件结构
- 固化知识图谱结构
- 明确旧文件到新文件的映射关系
- 明确哪些文件转为内部文件

产出：

- 更新 `SKILL.md`
- 更新 chapter output standard
- 新的文件模板
- `knowledge-map.json` 规范

### 阶段二：重写生成模板

目标：

- 用新的小节教学模板替换旧模板
- 加入概念复杂度分级
- 优先改 `detailed-notes.md` 的生成逻辑

优先章节：

- 树
- 图
- 栈/队列
- 串

因为这些最能暴露“只会整理、不会讲解”的问题。

### 阶段三：调整 HTML 渲染层

目标：

- 从“渲染 8 个文件”切换为“渲染 3 个主文件”
- 在章节首屏渲染知识图谱
- 强化分层阅读与任务模式

优先能力：

- 学习模式 / 复习模式
- 知识图谱节点跳转
- trace 展开
- 答案延迟显示
- 术语/公式卡片化

- 增加新的 P0/P1 检查

## 11. 旧结构到新结构的映射建议

### 10.2 不建议继续保留的学习入口

不建议继续让学习者直接面对：

- `learning-path`

- `code-extracts`
- `memory-cards`

这些更适合作为主文件里的“章节块”，而不是独立入口。


## 12. 推荐的新目录形态

建议未来章节目录结构调整为：

```text
chapters/<chapter-id>/
|-- detailed-notes.md
|-- practice.md
|-- review-notes.md
|-- knowledge-map.json
|-- source-map.md
`-- _meta.json
```

课程级仍保留：

```text
lessons/<source-id>/
`-- chapter-index.md
```

这套结构的好处是：

- 学习入口稳定且极少
- HTML 显示更容易设计
- 审校与机器元数据仍然保留


## 13. 重构优先级建议

若资源有限，建议按下面顺序推进：

1. 先定义三类主文件结构
2. 再定义知识图谱与复杂度分级
3. 再重写详细笔记的小节模板
4. 再把练习与复习合并
5. 最后调整 renderer 和质检脚本

原因：

- 内容模板不改，网页改再多也只是包装
- 文件结构不收敛，后续任何 UI 都会继续背负复杂输入


## 14. 结论

本次整改不应被理解为“把 Markdown 换成 HTML”，而应理解为：

1. 把章节产物从“碎文件包”收敛成“三类主文件”
2. 把内容从“整理笔记”升级成“可直接教学的讲义”
3. 把 HTML 从“文档阅读器”升级成“轻量网页教学层”

最终目标是：

- 学习者打开一个章节，不需要在 8 到 10 个文件之间做选择
- 学习者不需要回教材完成核心理解
- 同一套内容既可用于精学，也可用于复习
