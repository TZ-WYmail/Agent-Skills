# 知识点微讲与翻页式前端改造方案

## 1. 文档目标

本文件专门解决两个当前还没有被彻底解决的问题：

1. 现在的章节讲义虽然已经从 8 到 11 个文件收敛成了 3 个主文件，但“讲解单元”仍然偏大，很多知识点没有被真正讲透。
2. 现在的网页虽然已经有章节首屏、模式切换和知识图谱，但还没有真正变成“一个知识点一页”的教学阅读器。

这份方案的目标不是继续优化“章节摘要”，而是把 skill 的生成思路改成：

- 以知识点为真正的教学单元
- 以“10 到 15 分钟讲清一个知识点”为默认讲解尺度
- 以“翻页看知识点”作为前端主阅读模式


## 2. 对当前 vNext 的重新判断

当前 vNext 三主文件方向是对的，但它只解决了“章节入口过碎”的问题，还没有完全解决“知识点讲解不够像老师讲课”的问题。

当前仍然存在的不足：

- 讲解组织仍然偏“小节导向”，而不是彻底“知识点导向”
- 很多知识点虽然有结论、例子、trace，但解释密度仍然不够
- 某些知识点套了固定讲解骨架，导致输出结构稳定，但不一定是最适合初学者的解释顺序
- 前端当前主要还是“章节阅读页”，还不是“知识点翻页教学器”

因此，下一阶段不能只继续增强 `detailed-notes.md` 的段落结构，而要把“知识点页”提升为第一生产对象。


## 3. 新目标：章节是容器，知识点是核心页面

后续应把章节产物拆成两个层次：

### 3.1 学习者主入口

仍然保留三类主文件：

- `detailed-notes.md`
- `practice.md`
- `review-notes.md`

这是为了保留：

- Markdown 可审校性
- 章节级导出能力
- 打印 / 搜索 / 版本管理友好性

### 3.2 前端真正消费的细粒度教学单元

新增一个内部支持文件：

- `knowledge-pages.json`

它不取代三主文件，而是作为前端“翻页式知识点阅读器”的直接数据源。

换句话说：

- `detailed-notes.md` 是整章讲义汇编版
- `knowledge-pages.json` 是一页一知识点的前端教学版


## 4. 新的生成哲学：不是写小节，而是准备一段 10 到 15 分钟微讲

每个重要知识点在生成时，模型都应先回答这个问题：

`如果给我 10 到 15 分钟，只讲这一个点，我怎样讲才能让第一次学的人真正听懂？`

这会强迫模型不再满足于：

- 给结论
- 给一句直觉
- 给一个例子

而是要主动设计：

- 从哪里切入最容易懂
- 哪个例子最适合先讲
- 哪个误解最容易先出现
- 哪个“为什么成立”最关键
- 学完这页后学习者至少应该会什么

这就是后续知识点生成的第一原则。


## 5. 不再固定一个小节模板，而是固定“教学闭环要求”

后续不应继续强制所有知识点都使用同一套顺序完全相同的标题模板。

应改成：

- 固定必须完成的教学闭环
- 不固定闭环内部的呈现顺序

### 5.1 每个知识点页必须完成的最小教学闭环

无论是什么类型的知识点，至少都要让学习者读完后拿到这 6 件东西：

1. 这个知识点在解决什么问题
2. 这个知识点最核心的一句话是什么
3. 为什么这个说法 / 做法成立
4. 至少一个最小例子或最小反例
5. 最容易错在哪里
6. 闭卷时该怎样自己讲出来

### 5.2 不同类型知识点允许不同讲法

后续不要再默认每个点都强行按：

- 问题
- 结论
- 原理
- trace
- 易错点

这种统一顺序输出。

而应先识别知识点类型，再选讲解策略。


## 6. 知识点页的 5 类讲解模式

### 6.1 概念型

适用于：

- 定义
- 术语
- 基本性质

优先讲法：

- 先从现象或用途切入
- 再给精准定义
- 再给正例和反例
- 最后给边界提醒

### 6.2 结构表示型

适用于：

- 邻接矩阵
- 邻接表
- 十字链表
- 树 / 图 / 链表存储表示

优先讲法：

- 先说“为什么要这样存”
- 再用一张最小结构图或表
- 再说明字段各自的角色
- 再比较这种表示法牺牲了什么、换来了什么

### 6.3 过程算法型

适用于：

- DFS
- BFS
- Dijkstra
- Floyd
- Prim
- Kruskal

优先讲法：

- 先给起步场景
- 再说算法想维持的核心不变量
- 再跑最小 trace
- 再解释关键步骤为什么必要
- 最后指出最常见错法

### 6.4 对比辨析型

适用于：

- 连通分量 vs 强连通分量
- 生成树 vs 最小生成树
- BFS vs Dijkstra

优先讲法：

- 先给最容易混淆的地方
- 再用一张并排对比表
- 再给一题“看起来像 A 其实是 B”的判断例子

### 6.5 原理证明型

适用于：

- 为什么一次 DFS/BFS 恰好得到一个连通分量
- 为什么第一次发现边形成树边
- 为什么 Dijkstra 当前最短可锁定

优先讲法：

- 先讲错觉或疑问
- 再拆成 2 到 4 个关键论点
- 每个论点配一个极小例子
- 最后拼成完整解释


## 7. 10 到 15 分钟微讲的质量标准

每个 `high` 重要度且复杂度为 `C3/C4` 的知识点，应默认按“10 到 15 分钟微讲”产出。

学习结果至少要达到：

- 能自己复述
- 能判断相邻概念
- 能解释一个最小例子
- 能指出一个高频误区

如果某个知识点页读完后，学习者仍然需要回教材确认“它到底为什么成立”，则视为失败。


## 8. 新的数据结构建议

### 8.1 新增文件

每章新增：

- `knowledge-pages.json`

建议结构：

```json
{
  "chapter_id": "07-graph",
  "chapter_title": "第 7 章 图",
  "generated_at": "2026-06-13T20:00:00",
  "pages": []
}
```

### 8.2 每个知识点页的数据建议

```json
{
  "page_id": "kp-07-08",
  "knowledge_point_id": "kp-07-08",
  "title": "连通分量、生成树与生成森林",
  "type": "principle_explanation",
  "complexity_level": "C3",
  "importance_level": "high",
  "estimated_teaching_minutes": 12,
  "prerequisites": ["kp-07-06", "kp-07-07"],
  "learning_goal": "读完后能解释一次无向图遍历为什么恰好得到一个连通分量，并说明树边为什么构成生成树/森林",
  "entry_question": "遍历一次图，除了访问所有点，还会顺手暴露哪些结构信息？",
  "page_summary": "无向图中，从一个未访问点出发的一次 DFS/BFS 恰好覆盖它所在的一个连通分量；首次发现新点留下的树边连接全部访问点且不成环，因此形成生成树或生成森林。",
  "blocks": [],
  "practice_refs": ["Q-03", "Q-08"],
  "review_refs": ["C04", "模板 1"],
  "source_refs": ["PDF p.182-p.184"]
}
```

### 8.3 block 设计

`blocks` 不要求所有知识点一致，但建议支持这些类型：

- `hook`
- `intuition`
- `minimum_example`
- `formal_statement`
- `why_it_holds`
- `trace`
- `comparison`
- `confusion_fix`
- `implementation_bridge`
- `closed_book_retell`
- `mini_check`
- `recap`

重点在于：

- block 类型固定
- block 是否出现、顺序如何出现，由知识点类型决定


## 9. 生成流程重构

### 9.1 新的主流程

后续每章建议按这个顺序生成：

1. 抽取章节知识点
2. 建 `knowledge-map.json`
3. 给知识点分类、定复杂度、定重要度
4. 为每个知识点生成 `teaching brief`
5. 批量生成 `knowledge-pages.json`
6. 由知识点页汇编出 `detailed-notes.md`
7. 由知识点页映射生成 `practice.md`
8. 由知识点页映射生成 `review-notes.md`
9. 跑质量门

### 9.2 teaching brief

每个知识点在真正写正文前，应先生成一份内部 brief：

- 这个点最像哪一类教学模式
- 这个点适合从哪里切入
- 最小例子选哪个
- 必讲的“为什么成立”是哪一段
- 必修复的误解是什么
- 闭卷输出要让学生说到什么程度

有了 brief，正文才不容易滑回“模板填空”。


## 10. 并行生成策略

### 10.1 为什么适合并行

知识点页天然适合并行，因为每个页的任务边界很清晰：

- 输入相对独立
- 输出结构稳定
- 只需共享前置知识点和相邻点信息

### 10.2 建议的并行单元

每个知识点页独立生成一份草稿。

如果运行环境允许并行，可按批次并行：

- 每批 3 到 5 个知识点页
- 先生成主干点
- 再生成支撑点和扩展点

### 10.3 并行时每页必须共享的上下文

每个并行任务至少带上：

- 当前知识点的 source slice
- 本章知识图谱摘要
- 该知识点前置知识点
- 该知识点相邻易混点
- 该知识点目标读者
- 目标讲解时间预算

### 10.4 并行后的回收动作

并行生成后，必须统一做两件事：

1. 语气和难度校准
2. 跨页去重和跳转修正

否则前端翻页会出现重复解释或风格割裂。


## 11. 前端翻页式阅读器方案

### 11.1 首屏结构

章节页首屏建议变成：

1. 章节标题
2. 一句话章节主问题
3. 知识图谱
4. 推荐学习路径
5. “开始学习”按钮

点击后进入知识点翻页模式。

### 11.2 翻页模式的核心交互

每一页只展示一个知识点。

每页顶部固定显示：

- 知识点名称
- 复杂度
- 重要度
- 预计阅读 / 讲解时长
- 前置知识点
- 当前在推荐路径中的位置

每页底部固定显示：

- 上一页 / 下一页
- 回到知识图谱
- 对应练习
- 对应复习卡

### 11.3 页内结构

页内不是大段长文，而是卡片化块：

- 引入问题卡
- 最小例子卡
- 为什么成立卡
- trace 卡
- 易错点卡
- 闭卷输出卡

但不是每页必须全有，而是按知识点类型动态显示。

### 11.4 导航方式

建议支持：

- 点击知识图谱节点进入对应页
- 左右方向键翻页
- 触摸滑动翻页
- 上下展开 / 收起 trace 和补充块
- “只看主干知识点”过滤
- “复习模式”只显示 recap、易错点、闭卷输出


## 12. 三主文件与知识点页的关系

### 12.1 `detailed-notes.md`

改成“章节级汇编版讲义”，由知识点页按推荐学习路径拼接而成。

它适合：

- 通读
- 搜索
- 打印
- 审校

### 12.2 `practice.md`

不再只按章节顺序排题，而是显式引用知识点页：

- 这道题对应哪个知识点页
- 做错后先回哪一页

### 12.3 `review-notes.md`

不是正文压缩，而是知识点页的复习聚合版：

- recap
- 易错点
- 口头复述
- 记忆卡片


## 13. 质量门新增要求

后续 checker 应新增这些规则：

### P0

- 高重要度 `C3/C4` 知识点没有形成独立知识点页
- 知识点页没有说明“这个点解决什么问题”
- 知识点页没有最小例子或最小反例
- 需要原理解释的点没有 `why_it_holds`
- 过程型知识点没有 trace
- 页读完后仍需要回教材理解核心机制

### P1

- 知识点页过长但没有分块
- 知识点页之间跳转关系不清
- 同一误区在多页重复讲，但没有统一表述
- 练习与复习没有反向链接到知识点页


## 14. 分阶段实施建议

### Phase A：先定义数据结构

目标：

- 固化 `knowledge-pages.json` schema
- 固化页级 block 类型
- 固化页级元数据

### Phase B：改生成逻辑

目标：

- 先产知识点 brief
- 再产知识点页
- 最后汇编三主文件

### Phase C：改 renderer

目标：

- 首页知识图谱
- 知识点翻页阅读器
- 练习 / 复习跳转

### Phase D：改 checker

目标：

- 检查页级教学闭环
- 检查 `why_it_holds`
- 检查 trace
- 检查回链

### Phase E：先用第 7 章试跑

建议先拿这些知识点做试点：

- 图的存储结构选择
- DFS
- BFS
- 连通分量、生成树与生成森林
- 最小生成树
- 拓扑排序
- Dijkstra


## 15. 我对后续 skill 改造的明确建议

后续真正改 skill 时，优先级应该是：

1. 不要再优化“统一小节模板”
2. 先把“知识点页”定义出来
3. 让模型按“10 到 15 分钟微讲”生成每个重点知识点
4. 让 `detailed-notes.md` 从“第一生成对象”降为“汇编产物”
5. 让前端从“章节阅读器”升级成“知识点翻页器”

这会比继续打磨章节大纲式讲义，更接近你真正想要的教学效果。
