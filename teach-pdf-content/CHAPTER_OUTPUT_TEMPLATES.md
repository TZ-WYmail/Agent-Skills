# 章节三类主文件模板规范

## 1. 文档目标

本规范用于把 `teach-pdf-content` 的章节产物，从“多个功能文件并列生成”重构为“3 个学习主文件 + 少量内部支撑文件”。

它解决的不是单纯的文件合并问题，而是以下三个问题：

1. 章节内容必须从“整理教材”升级为“可直接教学”。
2. 小节组织必须从“目录导向”升级为“知识点导向”。
3. Markdown 必须从“长文本容器”升级为“可被网页精细渲染的语义化教学源文件”。

本规范与以下文档配合使用：

- `REFACTOR_IMPROVEMENT_PLAN.md`
- `KNOWLEDGE_MAP_SCHEMA.md`
- `references/chapter_output_standard.md`


## 2. 目标目录结构

每一章对学习者只暴露 3 个主文件：

```text
chapters/<chapter-id>/
|-- detailed-notes.md
|-- practice.md
|-- review-notes.md
|-- knowledge-map.json
|-- source-map.md
`-- _meta.json
```

说明：

- `detailed-notes.md`：第一次学习和真正理解。
- `practice.md`：自测、做题、诊断、查漏。
- `review-notes.md`：二刷、复习、考前压缩。
- `knowledge-map.json`：前端知识图谱与节点跳转数据源。
- `source-map.md`：供审校和回溯，不作为主学习入口。
- `_meta.json`：纯系统元数据。

禁止继续让学习者直接面对 8 到 11 个同级学习文件。


## 3. 重构后的章节生成总流程

每章生成流程必须调整为：

1. 确定章节范围与来源。
2. 提取候选知识点，不直接开始写正文。
3. 标记本章主线知识点、支撑知识点、扩展知识点。
4. 识别每个小节中“新出现的概念”。
5. 对新概念做复杂度分级 `C1-C4`。
6. 生成 `knowledge-map.json`。
7. 先并行生成 `knowledge-pages.json` 中的知识点页。
8. 再由页级内容汇编 `detailed-notes.md`。
9. 再根据知识点与掌握目标生成 `practice.md`。
10. 最后生成 `review-notes.md`。
11. 生成 `source-map.md` 与 `_meta.json` 作为内部支撑。

核心变化：

- 过去是“按文件类别拆写”。
- 现在是“按知识点建模，再按学习阶段组织输出”。


## 4. 统一设计原则

### 4.1 三类主文件的定位分工

三类文件必须分别承担不同任务，不能只是内容长短不同。

#### `detailed-notes.md`

职责：

- 把教材内容讲懂。
- 补足“为什么成立”“为什么这样设计”“为什么不那样做”。
- 提供最小例子、过程追踪、边界、误区、闭卷表达模板。

不应退化为：

- 摘要汇编
- 定义堆叠
- 过长的原文改写

#### `practice.md`

职责：

- 把“看懂”转化成“能输出、能做题、能判断、能迁移”。
- 围绕知识点出题，而不是围绕页面顺序堆题。

不应退化为：

- 单纯题目罗列
- 提示和答案混在一起
- 只有题没有诊断价值

#### `review-notes.md`

职责：

- 把整章压缩成高回忆密度复习材料。
- 帮学习者快速找回章节主线、易错点、必记点。

不应退化为：

- 重新复制详解
- 把所有内容再次压成密集长文


### 4.2 章节组织从“目录导向”改为“知识点导向”

以后不能默认把教材目录当作最终教学结构。

必须先识别：

- 本章核心概念
- 前置概念
- 算法 / 过程
- 公式 / 规则
- 对比辨析点
- 实现锚点
- 练习目标

然后再决定：

- 哪些小节按教材标题保留
- 哪些小节需要合并讲
- 哪些知识点必须单独展开
- 哪些知识点只需作为补充插入

判断标准是“学习者是否更容易学会”，不是“是否更忠于原目录排版”。


### 4.3 小节必须形成教学闭环

每个核心小节都必须闭环回答以下问题中的大部分：

1. 这一节为什么会出现在这里？
2. 它解决什么具体问题？
3. 先看一个最小例子会看到什么现象？
4. 正式定义 / 结论是什么？
5. 为什么这个结论成立？
6. 过程是怎么一步步发生的？
7. 它和相邻概念差在哪里？
8. 初学者最容易误解什么？
9. 闭卷时应怎样自己讲出来？

如果一个小节只有“结论 + 例子”，仍然属于不合格教学稿。


### 4.4 Markdown 必须变成“语义化源文件”

Markdown 不再只是用来堆文本，而要为前端提供稳定的语义结构。

因此每个主文件中的标题和区块必须可被程序识别，至少做到：

- 章节级标题稳定
- 小节级标题稳定
- 知识点表稳定
- “为什么成立 / Trace / 易错点 / 闭卷输出”这些区块可稳定定位
- 题目、答案、复习卡片可稳定映射到知识点

不要求立刻引入复杂 DSL，但必须使用稳定模板。


## 5. 知识点建模规则

### 5.1 知识点粒度

每个知识点应该是“一个可被独立掌握、复习、出题、定位”的单位。

建议粒度：

- 不是整章一个点，太粗。
- 不是一句话一个点，太碎。
- 通常是“一个核心概念 / 一个过程机制 / 一个稳定对比 / 一个实现关键点”。

合适示例：

- 图的连通分量
- DFS 遍历树边
- 生成树与生成森林的区别
- 邻接表与邻接矩阵的空间特征

不合适示例：

- “图这一章”
- “定义中的第三行描述”


### 5.2 新概念识别

每个小节必须识别：

- 本节主知识点
- 本节辅助知识点
- 哪些是本节第一次正式出现的新概念

只有新概念才强制触发复杂度评估与教学深展开。

已有概念再次出现时，不应重复写成长篇定义，而应做：

- 复用
- 对比
- 应用
- 强化


### 5.3 复杂度分级

对每个“新出现的概念”进行 `C1-C4` 分级。

#### `C1`

特征：

- 与已有知识高度相似
- 定义直观
- 状态变化少
- 不需要专门 trace

写法要求：

- 一句主问题
- 一句正式表述
- 一个最小例子
- 一个边界提醒

#### `C2`

特征：

- 需要与已有概念区分
- 有一定使用条件
- 题目中会单独考

写法要求：

- 问题
- 结论
- 最小例子
- 为什么这样定义
- 易混点
- 闭卷说法

#### `C3`

特征：

- 需要过程追踪
- 容易误解
- 依赖多个前置知识点
- 与结构或算法机制绑定

写法要求：

- 起步例子
- 现象观察
- 正式结论
- 为什么成立
- trace
- 边界 / 误区
- 相邻概念比较
- 实现或题型连接
- 闭卷输出模板

#### `C4`

特征：

- 高抽象度
- 多层依赖
- 只看定义很难懂
- 可能是整章难点或考试主难点

写法要求：

- 引入问题
- 最小反例或最小例子
- 分层讲解
- 至少一个完整 trace
- 为什么成立
- 为什么不能替换成别的方法
- 与多个相邻概念比较
- 题型映射
- 代码 / 公式 / 图示联动
- 闭卷表达模板
- 复习压缩版


### 5.4 复杂度评分建议

每个新概念可按以下维度做 `0/1/2` 评分：

- 新颖度
- 抽象度
- 依赖深度
- 动态性
- 易错性
- 教材解释不足度

映射建议：

- `0-2` -> `C1`
- `3-5` -> `C2`
- `6-8` -> `C3`
- `9-12` -> `C4`

这个评分主要服务生成逻辑，不必全部展示给学习者，但要保留在内部元数据或 `knowledge-map.json` 中。


## 6. `detailed-notes.md` 模板规范

### 6.1 文件角色

`detailed-notes.md` 是整章最重要的教学主文件。

目标不是“把教材内容写下来”，而是让学习者在不反复回教材的前提下理解本章的核心机制。


### 6.2 顶层结构

建议固定结构如下：

```markdown
# 第 X 章 标题

## 这章解决什么问题
## 先修提醒
## 本章知识地图
## 建议学习路径
## 起步例子 / 章节引子
## 本章主线
## 小节讲解
## 关键术语与公式速查
## 实现与代码连接
## 易混概念总对比
## 闭卷输出模板
## 10 分钟速读路线
```

说明：

- `本章知识地图` 需要与 `knowledge-map.json` 对齐。
- `建议学习路径` 至少给出：首次学习路径、复习路径。
- `小节讲解` 是正文主体。
- `关键术语与公式速查` 是压缩查阅区，不应放在最前面。


### 6.3 小节结构

每个核心小节建议采用以下骨架：

```markdown
### 7.4.1 连通分量、生成树与生成森林

#### 本节主问题

#### 本节知识点
| 知识点 | 是否新概念 | 复杂度 | 重要度 | 掌握目标 |

#### 先看一个最小例子

#### 先观察会发生什么

#### 正式结论 / 定义

#### 为什么成立

#### 过程追踪 / Trace

#### 边界、反例与易错点

#### 和相邻概念的区别

#### 代码 / 实现 / 题型连接

#### 本节闭卷输出模板

#### 本节自测
```

如果某小节是轻量概念，可按复杂度裁剪，但以下四块尽量保留：

- `本节主问题`
- `本节知识点`
- `正式结论 / 定义`
- `本节闭卷输出模板`


### 6.4 小节内部必须回答的关键问题

对于结构、算法、遍历、递归、状态变化类小节，必须额外回答：

- 为什么只会走到这些状态？
- 为什么不会漏掉应该覆盖的对象？
- 为什么不会重复计入错误结果？
- 为什么这一步是必要的？
- 为什么形成的是这个结构而不是别的结构？

如果讲“图遍历得到连通分量”，就必须解释：

- 为什么只会到达同一块内的顶点
- 为什么不会漏掉这一块中的点
- 为什么第一次到达新点留下的边不会成环
- 为什么这些边能把访问到的点全部连起来

这类解释缺失时，学习者仍然需要回教材或问老师，说明讲解失败。


### 6.5 `detailed-notes.md` 中的知识点表

每个核心小节必须有一个小型知识点表，用于支撑前端跳转与学习者定位。

建议字段：

| 字段 | 说明 |
| --- | --- |
| 知识点 ID | 对应 `knowledge-map.json` 节点 |
| 名称 | 知识点标题 |
| 是否新概念 | `是 / 否` |
| 复杂度 | `C1-C4` |
| 重要度 | `高 / 中 / 低` |
| 掌握目标 | `识别 / 解释 / 区分 / 复述 / 应用 / 诊断` |

作用：

- 明确本节到底在教什么
- 明确哪些点是第一次出现
- 支撑前端按复杂度和重要度筛选


### 6.6 `detailed-notes.md` 的语义化区块

为增强 Markdown 到网页的适配性，正文中应保持以下固定区块命名：

- `本节主问题`
- `先看一个最小例子`
- `正式结论 / 定义`
- `为什么成立`
- `过程追踪 / Trace`
- `边界、反例与易错点`
- `和相邻概念的区别`
- `代码 / 实现 / 题型连接`
- `本节闭卷输出模板`
- `本节自测`

这样前端后续可以把这些区块渲染成：

- 问题卡
- 例子卡
- 原理卡
- Trace 面板
- 易错警示卡
- 对比卡
- 代码联动卡
- 闭卷复述卡

也就是说，后面网页教学能力的增强，应建立在这套稳定语义结构上，而不是依赖自由文本猜测。


### 6.7 `detailed-notes.md` 的“闭卷输出模板”

每个核心小节必须给学习者一个“自己能讲出来”的模板。

模板的目的不是压缩原文，而是帮助学习者形成可复述结构。

建议样式：

```markdown
#### 本节闭卷输出模板

一句话版：

展开版：

如果题目问“为什么”应补充：
```

例如：

```markdown
一句话版：
在无向图中，从一个未访问顶点发起一次 DFS 或 BFS，访问到的恰好是它所在的一个连通分量。

展开版：
因为遍历只能沿已有边扩展，所以不会走到别的连通块；而同一连通块内任一点都存在从起点可达的路径，遍历会沿路径逐步访问到这些点，因此不会漏点。

如果题目问“为什么能得到生成树/森林”应补充：
第一次到达新顶点时保留的那条边不会形成环，但会把新顶点接到已有访问部分上，所以这些树边连通且无环；连通图得到生成树，非连通图得到生成森林。
```


### 6.8 `detailed-notes.md` 中的代码连接要求

如果本章存在实现价值，正文必须显式连接到代码层，而不能把代码完全丢到单独文件后就不管。

至少要回答：

- 这个概念在代码里对应什么数据结构？
- 空情况 / 边界情况是什么？
- 最容易写错的位置在哪？
- 学习者应该去看哪一段代码？

如果使用单独代码摘录，应在正文写明：

- 代码片段 ID
- 代码片段作用
- 阅读顺序


## 7. `practice.md` 模板规范

### 7.1 文件角色

`practice.md` 的任务不是“再给一堆题”，而是把知识点转成可被检查的输出能力。

它应覆盖：

- 闭卷解释
- 判断辨析
- 过程追踪
- 基础题
- 核心题
- 迁移题
- 常见错误诊断


### 7.2 顶层结构

建议固定结构如下：

```markdown
# 第 X 章 练习与自测

## 使用方式
## 知识点与题目映射
## 闭卷解释题
## 判断与辨析题
## 过程追踪题
## 基础练习
## 核心练习
## 迁移练习
## 常见错误诊断
## 完整答案与评分点
```


### 7.3 出题组织方式

题目优先按“知识点与能力目标”组织，而不是简单按教材页码顺序组织。

每道题都应标明：

- 题目 ID
- 对应知识点 ID
- 题型
- 难度
- 目标能力
- 建议耗时

示例：

```markdown
### Q-07

- 知识点：KP-07-04-01-A, KP-07-04-01-B
- 题型：过程解释
- 难度：中
- 目标能力：解释 + 区分
- 建议耗时：4 分钟

题目：

提示：
```


### 7.4 题型配比建议

每章至少覆盖以下四类：

#### 闭卷解释题

目标：

- 检查学习者能否不用看资料讲清楚一个核心点。

#### 判断与辨析题

目标：

- 检查学习者能否区分相近概念。

#### 过程追踪题

目标：

- 检查学习者是否真的理解状态变化，而不是只记结论。

#### 迁移题

目标：

- 检查学习者能否把概念放到新场景中用。

对于图、树、递归、算法章节，`过程追踪题` 不能缺席。


### 7.5 答案区规范

`practice.md` 的前半部分必须尽量无完整答案。

可以有：

- 一句提示
- 对应小节
- 相关知识点

不可以有：

- 直接给出核心结论
- 把答案写在题目下面

答案统一放在：

```markdown
## 完整答案与评分点
```

每道题答案至少包含：

- 标准答案
- 得分点 / 自查点
- 常见误答
- 错误原因
- 回看位置


### 7.6 `practice.md` 对前端的支撑

前端应能够基于该文件实现：

- 按题型切换
- 按知识点筛题
- 延迟显示答案
- 按难度筛题
- 聚合错题类型

因此题目元信息必须稳定存在。


## 8. `review-notes.md` 模板规范

### 8.1 文件角色

`review-notes.md` 是整章压缩后的高密度回忆入口。

它服务的是：

- 学完后的二刷
- 隔天 / 隔几天回顾
- 考前快速恢复主线

它不是简单把 `detailed-notes.md` 缩短，而是重新组织成“回忆优先”的结构。


### 8.2 顶层结构

建议固定结构如下：

```markdown
# 第 X 章 复习笔记

## 本章 3 分钟回忆路线
## 本章 10 分钟复习路线
## 必记定义 / 规则 / 结论
## 高价值易错点
## 概念对比速查
## 口头复述模板
## 记忆卡片
## 错题回炉规则
## 分日复习计划
```


### 8.3 回忆路线设计

至少提供两种路线：

#### 3 分钟回忆路线

适合：

- 刚学完后的快速回想
- 考前临时找回章节主线

内容应聚焦：

- 章节主问题
- 3 到 5 个主干知识点
- 最容易混淆的 1 到 3 个点

#### 10 分钟复习路线

适合：

- 一轮完整快速复习
- 从主线回到关键机制

内容应聚焦：

- 主线知识点顺序
- 必记定义 / 公式
- 高频 trace 或过程骨架
- 常错题型提醒


### 8.4 必记内容的选择标准

不是所有内容都应该进入复习压缩区。

优先进入：

- 课程要求能准确说出的定义
- 高频题型依赖的判断规则
- 稳定过程骨架
- 必须能区分的相邻概念
- 高频误区修正语句

不优先进入：

- 冗长解释段
- 只适合首次理解的铺垫性比喻
- 低频实现细节


### 8.5 口头复述模板

每章至少应给出若干“口头复述模板”。

作用：

- 帮学习者把被动阅读转成主动输出
- 帮前端做“口述模式”或“遮挡模式”

建议样式：

```markdown
### 口头复述模板 1

题目：

标准骨架：

必须提到：

容易漏掉：
```


### 8.6 记忆卡片组织

建议分为三组：

- `Core`：主干概念、定义、规则
- `Secondary`：支撑点、补充点
- `Trap`：高频误区、相似概念辨析

每张卡片必须映射到知识点 ID。


### 8.7 分日复习计划

建议固定节奏：

- 今天
- 明天
- 三天后
- 一周后

每个时点都要明确：

- 看什么
- 说什么
- 练什么
- 错了怎么办

避免空泛写法，如“回顾一下本章重点”。


## 9. 三类主文件之间的映射关系

### 9.1 同一知识点在三类文件中的角色不同

同一知识点应在三类文件中呈现不同层次：

- 在 `detailed-notes.md` 中，负责“讲懂”。
- 在 `practice.md` 中，负责“检验会不会用”。
- 在 `review-notes.md` 中，负责“快速想起来”。

不能直接复制同一段话到三个文件中。


### 9.2 知识点 ID 必须贯通

每个知识点至少要能映射到：

- `detailed-notes.md` 中的讲解位置
- `practice.md` 中的相关题目
- `review-notes.md` 中的复习位置
- `knowledge-map.json` 中的节点

这会直接决定前端是否能实现：

- 节点跳转
- 点到题
- 点到复习卡
- 模式切换


## 10. 与 `knowledge-map.json` 的对接要求

### 10.1 章节首屏必须有知识图谱入口

后续前端中，每章页面首屏建议顺序：

1. 章节标题
2. 章节主问题
3. 知识图谱
4. 推荐学习路径
5. 进入详细笔记

知识图谱不是装饰层，而是导航层。


### 10.2 三类主文件必须反向服务知识图谱

知识图谱不应该只从内容中“抽出”，也要反过来约束内容结构。

要求：

- `detailed-notes.md` 的小节要能被知识点节点定位
- `practice.md` 的题目要能被知识点节点聚合
- `review-notes.md` 的卡片和复述模板要能被知识点节点聚合


## 11. Markdown 到网页的适配增强策略

### 11.1 不建议直接放弃 Markdown

当前更优策略仍然是：

`教材 / PDF -> Markdown 语义化主文件 -> HTML 教学层`

原因：

- Markdown 易生成、易维护、易审校
- HTML 可以专注做教学交互
- 语义结构清晰后，网页增强成本更低


### 11.2 真正需要增强的是 Markdown 的“结构性”

当前问题不是“用了 Markdown 所以不好看”，而是：

- 内容结构对前端不友好
- 没有稳定语义块
- 没有知识点级映射
- 没有教学层级控制

因此整改方向是：

- 不是把 Markdown 换成 HTML 源文件
- 而是把 Markdown 升级成“可渲染的教学语义稿”


### 11.3 未来前端可利用的稳定语义

基于本规范，前端可逐步实现：

- 首屏知识图谱
- 本节主问题卡
- “为什么成立”高亮卡
- Trace 折叠展开
- 易错点警示条
- 概念对比卡片
- 口头复述模式
- 题目延迟答案模式
- 复习模式下的高价值节点高亮

所以前端优化的前提不是“多写 CSS”，而是“源内容先足够可结构化”。


## 12. 对现有 11 文件问题的整改规则

### 12.1 学习者入口必须收敛为 3 类

一个章节不应再生成 11 个并列学习文件。

原因：

- 学习者会在文件选择上消耗精力
- 前端会被迫处理碎片化输入
- 内容会自然向“功能模块拆散”倾斜，而不是形成完整教学流


### 12.2 旧文件到新文件的映射

建议统一映射为：

- `00-learning-path.md` -> `detailed-notes.md`
- `01-lesson-notes.md` -> `detailed-notes.md`
- `04-glossary.md` -> `detailed-notes.md`
- `07-code-extracts.md` -> `detailed-notes.md`

- `02-active-recall.md` -> `practice.md`
- `03-exercises.md` -> `practice.md`

- `05-review-plan.md` -> `review-notes.md`
- `06-memory-cards.md` -> `review-notes.md`

- `source-map.md` -> 内部支撑
- `_meta.json` -> 内部支撑
- 知识点建模结果 -> `knowledge-map.json`


## 13. 针对图论 / 数据结构章节的刚性要求

如果一个小节属于以下类型：

- 树
- 图
- 遍历
- 递归
- 状态变化
- 存储结构
- 算法过程
- 构造过程

则 `detailed-notes.md` 中至少必须包含：

1. 一个最小例子
2. 一个过程追踪
3. 一个“为什么成立”的解释
4. 一个易错点
5. 一个闭卷表达模板

缺其中任意一项，都应判为教学闭环不完整。


## 14. 示例：7.4.1 的目标改写方式

以“7.4.1 连通分量、生成树与生成森林”为例，未来不应只写：

- 本节回答什么问题
- 先给结论
- 原文依据
- 直觉理解
- 简单例子

而应补齐成：

```markdown
### 7.4.1 连通分量、生成树与生成森林

#### 本节主问题
一次图遍历除了“访问所有点”，还能顺便得到哪些结构信息？

#### 本节知识点
| 知识点 | 是否新概念 | 复杂度 | 重要度 | 掌握目标 |
| --- | --- | --- | --- | --- |
| 连通分量 | 是 | C3 | 高 | 解释 + 应用 |
| 生成树 | 是 | C3 | 高 | 解释 + 区分 |
| 生成森林 | 是 | C2 | 中 | 解释 |

#### 先看一个最小例子
给出一个分成三块的小图，让学习者先观察从某点出发能走到哪里、走不到哪里。

#### 先观察会发生什么
从 A 出发只能到达 A 所在那一块；换另一个未访问起点才会进入新的一块。

#### 正式结论 / 定义
给出连通分量、生成树、生成森林的正式表述。

#### 为什么成立
解释“不会跨块”“不会漏块内点”“第一次到达新顶点留下的边不会成环但会连通新顶点”。

#### 过程追踪 / Trace
给一个 DFS 或 BFS 的简短状态表，显示访问顶点集和树边集如何变化。

#### 边界、反例与易错点
说明它得到的是生成树，不是最小生成树；说明有向图不能直接照搬同一说法。

#### 和相邻概念的区别
生成树 vs 最小生成树；连通分量 vs 强连通分量。

#### 代码 / 实现 / 题型连接
说明在代码中通常如何记录 visited、parent、tree_edges。

#### 本节闭卷输出模板
给一句话版与展开版。
```

只有补到这一步，这一节才真正具备“教学可独立阅读”的能力。


## 15. 质量门建议

以下情况建议视为 P0：

- 章节仍生成大量并列学习文件
- 小节只给结论，不解释为什么成立
- 结构 / 算法类小节没有最小例子
- 结构 / 算法类小节没有 trace
- 学习者读完后仍必须回教材才能理解核心机制
- `practice.md` 的题目区直接露出完整答案
- `review-notes.md` 只是简化版正文，没有回忆导向结构
- 三类主文件与知识点 ID 没有贯通

以下情况建议视为 P1：

- 小节有解释，但没有明确易错点
- 有知识点表，但没有掌握目标
- 有复习内容，但没有分日节奏
- 有题目，但缺少错误原因诊断


## 16. 分步落地顺序

建议按以下顺序推进：

1. 先稳定三类主文件模板。
2. 再稳定知识点建模与复杂度分级。
3. 再稳定 page-first 的 `knowledge-pages.json` 生成逻辑。
4. 再把 `detailed-notes.md` 改为由页级内容汇编。
5. 再重写 `practice.md` 与 `review-notes.md` 的生成逻辑。
6. 最后让前端 renderer 对接新模板和 `knowledge-map.json`。

原因：

- 如果源内容结构不改，前端只是在美化半成品。
- 如果知识点映射不稳定，知识图谱和网页交互无法真正成立。

基于本规范，下一批可直接执行的任务建议为：

   - 闭卷输出卡片模式


## 18. 结论

这次重构的核心不是“把 Markdown 变成 HTML”，而是：

1. 把章节输出从碎片文件收敛成稳定的 3 类主文件。
2. 把内容从“摘要笔记”升级成“真正带讲解能力的教学稿”。
3. 把 Markdown 升级为可被前端精细教学化渲染的语义源文件。

最终目标是：

- 学习者打开一章就知道先看什么、学什么、练什么、复习什么。
- 学习者不必反复回教材补“为什么成立”。
- 前端不只是展示文档，而是真正承担“轻量网页教学层”的作用。
